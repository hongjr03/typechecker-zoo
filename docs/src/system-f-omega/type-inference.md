# 双向类型检查

类型推断引擎代表了System Fω实现中最关键的组成部分。它基于DK（邓菲尔德-克里希纳斯瓦米）工作列表算法，处理高阶多态、存在类型变量和双向类型检查之间的复杂交互，这使得System Fω既强大又难以高效实现。

该算法在一个**工作列表**上运行，其中包含判断和约束，系统性地将复杂的类型检查问题简化为更简单的问题，直到所有约束都被解决。这种方法为System Fω提供了可判定的类型推断，同时保持了主要类型并支持多态编程范式。

值得注意的是，这种方法与我们之前使用的直接树遍历方法不同。现在我们遍历语法树并发出约束，然后由工作列表算法求解这些约束，最后将结果反向传播回原始语法树以给出最终类型。这个系统允许引入更复杂的规则，但也使得整个过程变得“非局部”，因为我们是在处理由许多不同表达式生成的约束，因此错误报告变得更具挑战性，因为将约束问题回溯到其原始语法需要大量的簿记工作。

## 类型规则

现在，我们将大胆地进入下一代类型检查。我们的类型系统中引入了一些新符号，但它们大多与之前相同，只是增加了新的种类级操作。

* **\\( \\Gamma \\)（Gamma）**——类型上下文，存储变量及其类型的信息。
* **\\( \\vdash \\)（旋转门）**——“证明”关系，表示“给定此上下文，我们可以得出此判断”。
* **\\( \\Rightarrow \\)（双右箭头）**——综合模式，类型检查器在此模式下确定表达式的类型。
* **\\( \\Leftarrow \\)（双左箭头）**——检查模式，我们在此模式下验证表达式是否具有预期类型。
* **\\( \\forall\\alpha \\)（Forall Alpha）**——全称量化，表示“对于任何类型α”。
* **\\( :: \\)（双冒号）**——“具有种类”关系，告诉我们类型构造器属于哪个类别。
* **\\( \\star \\)（星号）**——具体类型（如`Int`、`Bool`、`String`）的种类。
* **\\( \\square \\)（方框）**——种类的种类，即种类本身所具有的种类。
* **\\( \\to \\)（箭头）**——项层面的函数类型，或类型层面的种类箭头。
* **\\( [B/\\alpha]A \\)** ——类型替换，将类型A中所有出现的类型变量α替换为类型B。
* **\\( \\Lambda\\alpha \\)（大Lambda）** ——类型抽象，在类型层面创建多态函数。
* **\\( \\kappa \\)（Kappa）** ——种类变量，表示我们类型系统中未知的类别。
* **\\( \\overline{x} \\)** ——上划线符号，表示元素序列或列表，例如多个变量或参数。

在检查DK工作列表算法的实现之前，让我们先建立支配System Fω的形式类型规则。这些规则通过高阶类型和更强大的多态性扩展了System F。

### 基本表达式规则

变量查找在上下文中找到类型：

\\[ \frac{x : A \in \Gamma}{\Gamma \vdash x \Rightarrow A} \text{(T-Var)} \\]

带双向检查的函数应用：

\\[ \frac{\Gamma \vdash e_1 \Rightarrow A \to B \quad \Gamma \vdash e_2 \Leftarrow A}{\Gamma \vdash e_1 \; e_2 \Rightarrow B} \text{(T-App)} \\]

检查模式下的Lambda抽象：

\\[ \frac{\Gamma, x : A \vdash e \Leftarrow B}{\Gamma \vdash \lambda x. e \Leftarrow A \to B} \text{(T-Lam)} \\]

### 类型应用与抽象

类型应用实例化多态表达式：

\\[ \frac{\Gamma \vdash e \Rightarrow \forall \alpha :: \kappa. A \quad \Gamma \vdash B :: \kappa}{\Gamma \vdash e \; [B] \Rightarrow [B/\alpha]A} \text{(T-TApp)} \\]

类型抽象引入了全称量化：

\\[ \frac{\Gamma, \alpha :: \kappa \vdash e \Leftarrow A}{\Gamma \vdash \Lambda \alpha :: \kappa. e \Leftarrow \forall \alpha :: \kappa. A} \text{(T-TAbs)} \\]

### 高阶种类规则

种类检查确保类型构造器是良好构成的：

\\[ \frac{}{\Gamma \vdash \star :: \square} \text{(K-Type)} \\]

\\[ \frac{\Gamma \vdash \kappa_1 :: \square \quad \Gamma \vdash \kappa_2 :: \square}{\Gamma \vdash \kappa_1 \to \kappa_2 :: \square} \text{(K-Arrow)} \\]

类型构造器应用：

\\[ \frac{\Gamma \vdash F :: \kappa_1 \to \kappa_2 \quad \Gamma \vdash A :: \kappa_1}{\Gamma \vdash F \; A :: \kappa_2} \text{(K-App)} \\]

### 模式匹配规则

构造器模式对照数据类型进行检查：

\\[ \frac{\Gamma \vdash e \Rightarrow T \quad \overline{\Gamma \vdash p_i \Leftarrow T \leadsto \Delta_i \quad \Gamma, \Delta_i \vdash e_i \Leftarrow A}}{\Gamma \vdash \text{match } e \text{ with } \overline{p_i \to e_i} \Rightarrow A} \text{(T-Match)} \\]

### 双向控制

从推断切换到检查的模式：

\\[ \frac{\Gamma \vdash e \Rightarrow A}{\Gamma \vdash e \Leftarrow A} \text{(T-Sub)} \\]

存在变量实例化：

\\[ \frac{\Gamma, \hat{\alpha} :: \kappa \vdash e \Rightarrow A}{\Gamma \vdash e \Rightarrow [\hat{\alpha}/\alpha]A} \text{(T-InstL)} \\]

在这些规则中，\\( \Rightarrow \\) 表示**推断**模式（综合类型），\\( \Leftarrow \\) 表示**检查**模式（对照预期类型进行验证），\\( :: \\) 将类型与其种类关联起来，而 \\( \hat{\alpha} \\) 表示在推断过程中求解的存在类型变量。

## DK 工作列表算法

DK算法将类型推断表示为约束求解问题。它不是递归遍历表达式并立即做出类型检查决策，而是将约束累积到工作列表中并系统性地处理它们。

### 工作列表结构

```rust
#![enum!("system-f-omega/src/worklist.rs", WorklistEntry)]
```

工作列表包含三种基本条目，它们协同工作以管理类型检查过程。类型变量绑定跟踪类型变量及其当前状态，可以是全称的（来自显式量词）、存在的（表示需要推断的未知类型）或已求解的（通过约束求解确定的存在变量）。项变量绑定将程序变量与其推断或声明的类型关联起来，维护源代码中标识符与其类型信息之间的映射。判断表示尚未完成的类型检查和推断任务，编码了例如子类型检查、类型综合和类型验证等操作。

该三元结构允许算法推迟复杂决策，同时系统性地积累所需信息以正确做出决策。通过将变量管理与实际的类型检查工作分离，算法能够处理复杂的依赖关系，并确保所有约束按正确顺序得到解决。

### 类型变量种类

```rust
#![enum!("system-f-omega/src/worklist.rs", TyVarKind)]
```

工作列表中的类型变量可以具有不同的状态，这些状态反映了它们在类型检查过程中的角色。通用类型变量表示由源代码中显式量词产生的普通类型变量，例如 \\( \forall\alpha.\ldots \\) 中的 \\( \alpha \\) 。这些变量具有固定作用域，表示抽象类型，可以在多态函数应用时用具体类型实例化。存在类型变量（记作 \\( \hat{\alpha} \\) ）表示需要通过约束求解来推断的未知类型。当算法遇到类型不立即显现的表达式时，会生成这些变量，并通过统一和约束传播来确定其类型。标记变量（记作 \\( \triangleright\alpha \\) ）充当作用域标记，当离开其作用域时，能够对类型变量进行垃圾回收。

存在变量的解存储在一个单独的映射中，而不是存储在工作列表条目本身中。当存在变量被求解时，会记录映射 \\( \hat{\alpha} = \tau \\) ，表明未知类型 \\( \hat{\alpha} \\) 已被确定为具体类型 \\( \tau \\) 。这种分离确保即使在工作列表条目经过处理并在约束求解过程中被移除后，解仍然可访问。

### 判断类型

```rust
#![enum!("system-f-omega/src/worklist.rs", Judgment)]
```

判断类型表示驱动 DK 算法前进的核心类型检查任务。子类型判断（记作 \\( A \leq B \\) ）用于验证类型 \\( A \\) 是否为类型 \\( B \\) 的子类型，这对处理函数类型和多态实例化中出现的逆变和协变关系至关重要。推断判断（记作 \\( e \Rightarrow A \\) ）为给定表达式 \\( e \\) 综合出一个类型 \\( A \\) ，允许算法在没有期望类型的情况下从表达式确定类型。检查判断（记作 \\( e \Leftarrow A \\) ）用于验证给定表达式 \\( e \\) 是否符合期望类型 \\( A \\) ，当目标类型已知时，这通常比综合模式更高效。

实例化判断处理多态类型实例化的复杂过程，管理在将多态函数应用于参数时产生的存在变量的创建和求解。应用判断为函数应用推断提供专门处理，处理当函数类型可能不立即已知时产生的复杂约束生成。算法通过模式匹配这些判断的结构，并根据需要生成新的工作列表条目来处理它们，从而创建一种系统性的约束求解方法，确保所有类型关系都得到正确建立。

## 双向类型检查

DK 算法采用双向类型检查，将推断分为两种互补模式，共同应对 System Fω 的复杂性。

### 综合模式（⇒）

```rust
#![function!("system-f-omega/src/worklist.rs", DKInference::solve_inference)]
```

综合模式通过分析表达式来确定其类型，从表达式的结构出发产生类型信息。当遇到变量时，算法会在当前类型上下文中查找其类型，并根据需要实例化任何多态模式，以生成当前使用场景的具体类型。函数应用要求算法推断函数和参数的类型，然后统一它们以确保应用是良好类型的。类型应用使用具体类型参数实例化多态类型，这需要仔细管理类型变量的替换和作用域。

类型注解为推断过程提供显式指导，允许程序员指定期望类型，算法可以利用这些类型来约束搜索空间。综合模式不仅生成一个类型，还生成任何为使类型有效而必须成立的约束，使算法能够处理被分析表达式不同部分之间的复杂依赖关系。

### 检查模式（⇐）

```rust
#![function!("system-f-omega/src/worklist.rs", DKInference::solve_checking)]
```

检查模式验证表达式是否符合期望类型，从已知类型信息反向工作以验证表达式结构。对于 lambda 表达式，算法可以立即分解期望的函数类型以提取参数和返回类型，然后对照期望的参数类型检查 lambda 参数，并递归检查函数体是否符合期望的返回类型。当对照多态类型进行检查时，算法会为量化变量引入新的类型变量，并对照实例化后的类型检查表达式。

当直接检查策略不适用时，算法会回退到综合加子类型的方法，即先为表达式综合出一个类型，然后验证这个综合出的类型是否为期望类型的子类型。检查模式通常比综合模式更高效，因为它可以对期望类型结构做出更强的假设，从而允许算法剪枝搜索空间并做出更有针对性的推断。

## 存在类型变量

存在变量表示使类型推断成为可能的未知类型。它们充当占位符，通过统一和约束传播来求解。

### 变量生成

```rust
#![function!("system-f-omega/src/worklist.rs", Worklist::fresh_evar)]
```

新的存在变量会以唯一的名称生成并添加到工作列表中。算法确保每个未知类型都有一个不同的变量，防止意外统一。

### 约束生成

当算法遇到具有未知类型的表达式时，会遵循系统性的约束生成过程。算法创建新的存在变量来表示未知类型，确保每个未知都有一个唯一的占位符，可以在求解过程中进行跟踪。然后，这些存在变量通过约束生成与已知类型关联，创建方程和不等式，以捕获程序结构所隐含的类型关系。

生成的约束被添加到工作列表中进行系统性求解，使算法能够推迟复杂的求解决策，直到获得足够信息。随着约束逐步解除且存在变量获得具体解，这些解会传播到整个约束系统，可能以级联效应促成更多约束的求解，从而逐步确定所有未知类型。

### 求解过程

```rust
#![function!("system-f-omega/src/worklist.rs", DKInference::solve)]
```

约束求解通过系统性的模式匹配过程进行，该过程确定工作列表中每个判断适用的约束求解规则。算法检查约束的结构，并应用适当分解策略将复杂约束拆分为更简单、更易于管理的部分。例如，涉及函数类型的约束可被分解为参数类型和返回类型的独立约束。

代入发挥着关键作用：已求解的变量会应用于整个约束系统，将其存在变量替换为已经确定的类型。发生检查（occurs check）通过确保类型变量不出现在自身求解结果中，防止在合一过程中创建无限类型，从而维护类型系统的健全性。求解过程保证终止，因为 System Fω 具有可判定的类型推断，这意味着每个良类型程序最终都会达到所有约束被求解、所有存在变量都有具体解的状态。

## 子类型与多态实例化

System Fω 的子类型关系体现了更普遍的多态类型是较不普遍多态类型的子类型这一思想。这使得多态函数的灵活使用成为可能。

### 子类型规则

```rust
#![function!("system-f-omega/src/worklist.rs", DKInference::solve_subtype)]
```

在比较类型是否满足子类型关系之前，算法执行一个关键步骤——**zonking**：将所有已求解的存在变量替换为其解。这确保子类型比较看到的是完全解析后的类型，而不是陈旧的变量名。如果没有 zonking，类似 \\( \\text{Bool} \\leq \\hat{\\alpha}_1 \\)、\\( \\hat{\\alpha}_1 \\leq \\hat{\\alpha}_0 \\)、\\( \\hat{\\alpha}_0 \\leq \\text{Int} \\) 这样的约束链永远无法检测出 \\( \\text{Bool} \\leq \\text{Int} \\) 的矛盾，因为每次比较都会看到未求解的变量名而非已经解析的类型。

子类型算法处理若干控制 System Fω 中类型兼容性的基本关系。自反性确保每个类型被视作自身的子类型，为子类型推导提供基础情况。传递性允许子类型关系通过中间类型进行组合，因此若 \\( A \\leq B \\) 且 \\( B \\leq C \\)，则自动有 \\( A \\leq C \\)。

函数类型呈现出经典的逆变-协变模式：函数类型 \\( A_1 \\to B_1 \\) 是 \\( A_2 \\to B_2 \\) 的子类型，当且仅当 \\( A_2 \\leq A_1 \\)（参数位置逆变）且 \\( B_1 \\leq B_2 \\)（结果位置协变）。全称量化遵循以下原则：\\( \\forall\\alpha. A \\leq B \\) 成立，当 \\( \\alpha \\) 被实例化为一个新的存在变量时 \\( A \\leq B \\) 成立。存在实例化要求算法以满足子类型约束的方式求解存在变量，这通常会导致创建需要被求解的额外约束。

### 多态实例化

```rust
#![function!("system-f-omega/src/worklist.rs", DKInference::instantiate_left)]
```

实例化通过系统性地管理全称量化与存在量化之间的关系，处理应用多态函数的复杂过程。算法首先识别多态类型中的量化变量，分析 \\( \\forall \\)-类型的结构以确定哪些类型变量需要实例化。对于每个量化变量，算法生成新的存在变量，这些变量将作为占位符，等待通过约束求解确定具体类型。

代入阶段将量化变量替换为整个类型中对应的存在变量，从而有效地将多态类型模式转换为具有未知组件的具体类型。添加额外的约束以确保正确的实例化，捕获实例化类型与多态函数使用上下文之间必须成立的关系。这一系统性过程使得同一个多态函数能够在不同上下文中以不同类型被使用，同时保持类型安全，并确保所有实例化与函数的类型签名一致。

## 错误处理与诊断

DK 算法通过跟踪约束的来源以及导致类型检查失败的推理过程，提供精确的错误报告。

```rust
#![enum!("system-f-omega/src/errors.rs", TypeError)]
```

错误系统涵盖了类型检查期间可能发生的几类失败。合一失败出现在算法试图使两个类型相等但发现它们根本不相容时，例如试图将 \\( \\text{Int} \\) 与 \\( \\text{Bool} \\to \\text{String} \\) 合一。发生检查违规通过检测类型变量是否会出现在其自身的解中（例如试图求解 \\( \\hat{\\alpha} = \\hat{\\alpha} \\to \\text{Int} \\)）来防止创建无限类型。

作用域违规发生在类型变量逃出其预期作用域时，典型情况是某个本应局限于特定约束求解上下文的存在变量出现在最终类型中。种类不匹配发生在类型构造器被错误应用时，例如将期望高阶类型参数的构造器应用于具体类型。每个错误都包含位置信息和失败的类型检查规则描述，从而实现精确的诊断，帮助程序员理解和修复代码中的类型错误。

当算法成功时，它会为每个表达式找到最一般的类型。这意味着用户能在不损失类型安全的前提下获得最大的多态性。

## 工作列表优化

还可以添加一些优化来提高此过程的效率。

```rust
#![function!("system-f-omega/src/worklist.rs", DKInference::solve_judgment)]
```

智能调度工作列表条目通过策略性约束排序来提高性能。该算法优先处理可以立即求解的简单约束，例如自反子类型关系或具体类型的直接统一，从而快速解决这些琐碎情况，并可能简化其他约束。需要大量推理或依赖其他约束求解的复杂约束会被推迟，直到通过求解较简单约束获得更多信息。

相关约束被批量处理以减少重复工作，使算法能够一次性求解一组相互依赖的约束，而无需多次重新处理。这种调度策略通过确保每个约束在最有可能被求解时进行处理，显著降低了约束求解的整体计算成本。

约束在加入工作列表之前会经过系统性的简化，以提高性能和清晰度。算法消除类似 \\( A \leq A \\) 这类始终满足的琐碎约束，避免不必要的工作并减少约束系统中的杂乱。尽可能合并相关约束以减小工作列表的整体规模，例如将涉及相同类型的多个子类型约束合并为更全面的关系。

简化过程还能早期检测不可满足的约束，使算法能够立即报告类型错误，而不是继续注定失败的约束求解尝试。这一预处理步骤确保工作列表中仅包含对类型推断过程有意义的约束，从而提高整体算法的效率，并使生成的错误信息更加精确。
